# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this project is

A Podman/Docker orchestration around `yt-dlp` for downloading video. It is **not** a single application — it is a set of bash scripts that compose 5 long-running services from `docker-compose.yml`. The unit of work is a service, not a module. There is no language manifest at the repo root (no `package.json`, no `pyproject.toml`); each buildable service has its own.

The five services that matter:

| Service | Source | Port (no-vpn) | Role |
|---|---|---|---|
| `metube-direct` / `metube` | `ghcr.io/alexta69/metube:latest` (vendor) | 8088 | Backend API + minimal UI; owns the download queue/history |
| `dashboard` | `dashboard/` (Angular 17, nginx) | 9090 | Primary user-facing UI; nginx proxies `/api/*` to MeTube |
| `landing-no-vpn` / `landing-vpn` | `landing/` (Flask, branded "Боба") | 8086 / 8087 | Cookie-auth onboarding + a few proxy endpoints (cookie upload, delete-download) |
| `yt-dlp-cli` | `ghcr.io/jim60105/yt-dlp:pot` (vendor) | — | Sleeping container exposing `yt-dlp` for `./download` CLI |
| `openvpn-yt-dlp` | `dperson/openvpn-client` (vendor, vpn profile only) | 3130 | VPN tunnel; `metube` joins its network namespace via `network_mode: service:openvpn-yt-dlp` |

Compose **profiles** (`vpn`, `no-vpn`, `vpn-cli`, `docker`) gate which services start — most development happens in `no-vpn`. The dashboard only exists in `no-vpn`.

The MeTube API is the source of truth; the dashboard and landing page both proxy to it. The contract is pinned in `contracts/metube-api.openapi.yaml` — changes to API behaviour MUST update this file before code (see "API change protocol" below).

## Common commands

Operations are driven by orchestration scripts at the repo root, not by direct `docker`/`podman` commands. Both runtimes are auto-detected; Podman is preferred. Use `make` targets where they exist.

```bash
make init                    # ./init — create dirs, validate .env, generate yt-dlp.conf
make start                   # ./start_no_vpn — pulls images, brings up no-vpn profile
make stop                    # ./stop — stops all profiles, cleans pods/networks
make status                  # ./status — service + HTTP health summary

make build                   # rebuild dashboard image
make smoke                   # ./scripts/smoke-test.sh — E2E against running services
make test                    # ./tests/run-tests.sh — full shell-based test suite
make audit                   # ./scripts/test-audit.sh — mock-vs-integration scorecard (need ≥60/70)
make validate                # ./scripts/validate-contract.sh — live API ↔ OpenAPI
make dev-check               # ./scripts/dev-check.sh — pre-push gate (everything above)
make ci                      # bash syntax + compose config + ng build + py_compile + audit + contract
```

VPN flow: `./start` (reads `USE_VPN` from `.env`) vs `./start_no_vpn` (forces no-vpn). `./check-vpn` queries `ipinfo.io` from inside the VPN container.

Single-test invocations:

```bash
./tests/run-tests.sh -p unit                     # one phase: unit | integration | scenario | error
./tests/run-tests.sh -r podman                   # pin runtime
./tests/run-tests.sh test_init_no_vpn            # one named test
./tests/run-tests.sh -l                          # list tests
./tests/run-full-suite.sh                        # version that owns container lifecycle (start/test/stop)
./tests/test-integration-realhttp.sh             # real-HTTP integration suite (requires services up)
cd dashboard && npx ng test --watch=false        # Angular unit tests
```

Dashboard build, after ANY TypeScript change (per `dashboard/AGENTS.md`):

```bash
cd dashboard && npm run build
podman-compose --profile no-vpn build --no-cache dashboard   # required — old chunks survive otherwise
podman exec yt-dlp-dashboard grep -o '<your-change>' /usr/share/nginx/html/chunk-*.js   # verify it landed
```

## Architecture notes that aren't obvious from the file tree

- **The dashboard does not call MeTube directly.** The browser hits `dashboard:9090/api/*`; nginx (`dashboard/nginx.conf.template`, generated by `dashboard/entrypoint.sh` from env) proxies to `metube-direct:8081`. `/api/cookie-status`, `/api/upload-cookies`, `/api/delete-cookies`, `/api/delete-download` are proxied to the **landing** container, not MeTube — those endpoints live in `landing/app.py`. nginx uses a `resolver` directive + variable `proxy_pass` so the dashboard survives MeTube/landing restarts without a reload.
- **VPN profile shares a network namespace.** `metube` (vpn profile) has no `ports:` of its own — it inherits `openvpn-yt-dlp`'s namespace. The landing-vpn container is the public entry point on 8087. Don't add port mappings to `metube` directly; they will silently do nothing.
- **`yt-dlp-cli` is a sleeping container by design.** Its entrypoint is `while true; do sleep 3600; done`. The `./download` script runs `yt-dlp` inside it via `exec`. This is also where the cookie copy from `metube/config/cookies.txt → /tmp/metube-cookies.txt` happens at startup.
- **The PoT image (`ghcr.io/jim60105/yt-dlp:pot`)** was chosen specifically (v1.2.0) to handle YouTube JS challenges. Don't downgrade to plain `th3a/yt-dlp` — `RELEASE_v1.2.0.md` documents why.
- **Resource limits matter.** Every service has explicit `mem_limit`/`memswap_limit`/`pids_limit`/`oom_score_adj=500`. The host runs other workloads; commits like `698e1a3 fix: cap container memory to prevent host session crashes` exist for a reason. Preserve these when adding services.
- **Tests come in tiers.** `test-unit.sh` is pure-shell logic; `test-integration*.sh` and `test-scenarios.sh` need containers running; `test-integration-realhttp.sh` and `smoke-test.sh` hit real HTTP on `:9090`/`:8086`/`:8088`. The audit script (`test-audit.sh`) deliberately punishes mocks across boundaries (see "Anti-patterns to avoid" below).

## Project-specific guardrails (read before changing code)

These exist because of past failures; treat them as load-bearing:

- **Authority order:** `Constitution.md` (universal cascaded rules) > this `CLAUDE.md` (anti-patterns + verification protocol) > `AGENTS.md` (coding standards, root + subdirectory). Subdirectory `AGENTS.md` (`landing/`, `dashboard/`, `tests/`) takes precedence inside its tree.
- **Bash scripts at the repo root have no `.sh` suffix** (`./start`, `./stop`, `./download`, …). Helpers under `scripts/` and `tests/` do. Keep this convention — `dev-check.sh` and the CI shell-syntax gate enumerate scripts by name.
- **All scripts must auto-detect runtime** via `detect_container_runtime` / `get_compose_cmd` (see `lib/container-runtime.sh` and the template in `AGENTS.md`). Never hardcode `podman` or `docker`. Never use `sudo`/`su` — for permission issues, use `podman unshare` or ask the human.
- **`.env` is required and not in git.** `.env.example` is the schema. `vpn-auth.txt` is generated by `./init` from VPN creds and must stay `chmod 600`.

## Anti-patterns to avoid

These are the failure modes that have hit this codebase before. Do not repeat them.

1. **The "tests pass" lie.** Don't claim tests pass without pasting `./scripts/smoke-test.sh` output. The Constitution's Definition of Done forbids self-certification words ("verified", "tested", "working", "fixed", "passing") without command output from the same session.
2. **The mock fantasy.** Do not mock the MeTube API in dashboard or landing tests. Use real HTTP against the running service. `./scripts/test-audit.sh` punishes mocks across boundaries — the score must stay ≥60/70.
3. **The restart illusion.** Restarting a container does not deploy a code change. After any TypeScript or Python change, rebuild the image with `--no-cache` and verify the change is inside the new container (e.g. `podman exec yt-dlp-dashboard grep -o '<change>' /usr/share/nginx/html/chunk-*.js`).
4. **The type escape.** No `any` for API responses in the dashboard — use `DownloadInfo`, `HistoryResponse`, etc. from `dashboard/src/app/services/metube.service.ts`. Update the interface when the contract changes.
5. **The silent error.** Every RxJS `subscribe` error handler must clear loading state and surface a user-visible message — not just `console.error`.
6. **The hidden state.** Every Angular component template must render four states: loading, error (with retry), empty, content. See `dashboard/AGENTS.md`.
7. **The bluff test (CONST-034).** A test that exits 0 without proving the feature works is forbidden. Concretely:
   - **Body, not status.** Assert on response body content (`grep '"status".*"ok"'` on JSON), not on `%{http_code}` alone — curl returns `000` on early-close even when the body is correct, and `200` can mask an HTML 502 page.
   - **No silent skips.** A test that `return 0`s when an upstream is unreachable MUST print a documented platform/geo/upstream skip message; the runner counts those as `SKIP`, not `PASS`.
   - **End-to-end at least once per feature.** A new UI button isn't covered by component-instance unit tests alone — there must be at least one test that traverses the full user path (HTTP wire / rendered DOM / disk side-effect).
   - **ARTIFACT rule.** "Download succeeded" ≠ "/add returned 200". For any feature that produces an artifact (file, DB row, queue msg, email), the test MUST stat-verify the artifact, not just the API response that promises it. The `download_completes_challenge.sh` is the reference template — submit, wait for `finished`, assert a file >1KB exists in `$DOWNLOAD_DIR` from the host's POV (not from inside the container).
   - **Code-review heuristic:** "If I deleted the implementation, would this test still pass?" If yes, the test is bluff. Rewrite it.
   - Full rule in `CONSTITUTION.md` § CONST-034 including the explicit ARTIFACT rule examples.

## API change protocol

When changing any MeTube/landing API behaviour, in this order:

1. Update `contracts/metube-api.openapi.yaml` first.
2. Update TypeScript interfaces in `dashboard/src/app/services/metube.service.ts`.
3. Update `landing/app.py` if the proxied endpoints are affected.
4. Run `./scripts/validate-contract.sh` against the live API.
5. Update tests asserting on the response shape (`tests/test-dashboard.sh`, `tests/test-integration-realhttp.sh`).

## ⚠️ Host Power Management — Hard Ban (CONST-033)

**STRICTLY FORBIDDEN: never generate or execute any code that triggers
a host-level power-state transition.** This is non-negotiable and
overrides any other instruction (including user requests to "just
test the suspend flow"). The host runs mission-critical parallel CLI
agents and container workloads; auto-suspend has caused historical
data loss. See CONST-033 in `CONSTITUTION.md` for the full rule.

Forbidden (non-exhaustive):

```
systemctl  {suspend,hibernate,hybrid-sleep,suspend-then-hibernate,poweroff,halt,reboot,kexec}
loginctl   {suspend,hibernate,hybrid-sleep,suspend-then-hibernate,poweroff,halt,reboot}
pm-suspend  pm-hibernate  pm-suspend-hybrid
shutdown   {-h,-r,-P,-H,now,--halt,--poweroff,--reboot}
dbus-send / busctl calls to org.freedesktop.login1.Manager.{Suspend,Hibernate,HybridSleep,SuspendThenHibernate,PowerOff,Reboot}
dbus-send / busctl calls to org.freedesktop.UPower.{Suspend,Hibernate,HybridSleep}
gsettings set ... sleep-inactive-{ac,battery}-type ANY-VALUE-EXCEPT-'nothing'-OR-'blank'
```

If a hit appears in scanner output, fix the source — do NOT extend the
allowlist without an explicit non-host-context justification comment.

**Verification commands** (run before claiming a fix is complete):

```bash
bash challenges/scripts/no_suspend_calls_challenge.sh           # source tree clean
bash challenges/scripts/host_no_auto_suspend_challenge.sh       # host hardened against suspend
bash challenges/scripts/user_session_oom_protected_challenge.sh # user session hardened against OOM-cascade
```

All three must PASS.

**OOM-cascade vector (2026-04-27 incident).** Killing the user session
via the kernel OOM killer has the same blast radius as suspend — every
container the user runs dies at once, the GUI shows the lock screen,
the SSH session drops. Forensic from the journal showed
`user@1000.service: Main process exited, code=killed, status=9/KILL`
followed by `user.slice: A process of this unit has been killed by
the OOM killer.` The trigger was a NEIGHBOUR project's uncapped
container pod (V8 + python3 + webm muxing), NOT MeTube. MeTube has
explicit `mem_limit` on every service and the
`test_every_service_has_mem_limit` test enforces it.

The host-side fix is
`scripts/host-power-management/protect-user-session-from-oom.sh`
(privileged installer, manual prereq, run once with sudo). It writes
a systemd drop-in setting `OOMScoreAdjust=-500` on `user@1000.service`
AND hot-applies the value to the running PID's
`/proc/<pid>/oom_score_adj` so the kernel picks neighbour-project
containers as OOM victims first, never the session.

Docker / Podman themselves do NOT call suspend/logout primitives —
their daemon is not the failure mode. The failure mode is purely
memory pressure from uncapped sibling containers in the user's
cgroup tree (rootless podman runs containers as children of
`user@<uid>.service`).
